## 1. Retrieval Context Delivery (shipped)

### Problem
The planning pipeline sent `raw_summaries` (~9.4K tokens) to reasoning calls — full structural index + inline seed file source. This limited file count to ~30 at 32K context.

### Experiments (A/B tested, 10 runs each, temp=0)

| Approach | Prompt Size | Quality | Result |
|----------|------------|---------|--------|
| Full index + seeds (baseline) | 9.4K tok | 10/10 consistency | — |
| One-line manifest (tiered) | 4.3K tok | 0/10 openai_provider signal | **Failed** — lost import-chain awareness |
| Two-line manifest (doc + classes) | 5.4K tok | 12 approaches instead of 6 | **Failed** — hallucinated |
| No seeds, full index | 4.5K tok | 10/10, improved wrapper signal | **Shipped** — seeds were noise |
| Manifest + inspect_files tool | 4.4K tok | 10/10, 40% faster | **Shipped** — best of both |

### Key finding
**The structural index (with imports) is load-bearing.** Trimming it loses signal that changes architecture decisions. But inline seed source is noise — 5K tokens the model ignores. The `inspect_files` tool lets the model fetch structural detail on demand.

### Shipped changes
- `raw_summaries` now contains one-liner manifest (path + docstring) instead of full index
- `inspect_files(paths)` tool serves structural detail on demand (classes, methods, imports)
- `read_files` capped at 10 per call, `inspect_files` at 15 — prevents context blowup on dense models
- Default file count bumped from 30 to 50

## 2. Retrieval Quality (code path)

### 30B MoE vs 4B for retrieval

| Model | Critical Recall | Perfect | Avg Time |
|-------|----------------|---------|----------|
| Qwen3.5-4B | 78% | 17/40 | 22s |
| **Qwen3-Coder-30B** | **89%** | **28/40** | **18s** |

The 30B MoE is both better AND faster than the 4B (MoE generates faster). Hybrid model split eliminated — single model for everything.

### File count experiments

| Files | Critical Recall | Perfect | Avg Time |
|-------|----------------|---------|----------|
| 30 | 89% | 28/40 | 18.1s |
| 50 | 88% | 25/40 | 18.7s |
| 60 | 90% | 29/40 | 36.2s |
| 100 | 89% | 27/40 | 55.6s |

**30 files is the sweet spot.** More files dilute rather than help — the remaining 11% is LLM scan non-determinism, not file count.

### Pipeline fix attempts (all reverted)
- Depth-2 import expansion: router.py went from 0 to 3 misses
- Path-based scoring + enhanced scan prompt: no reliable improvement
- Both reverted — baseline at 30 files was already near-optimal

### Structural gap analysis
The consistently missed files follow patterns:
- Small definition files (enums, protocols) with minimal structural footprint
- Registry/factory-loaded files (dynamic import, invisible to AST)
- Files reachable only at import depth 2+
- These are deterministic signal gaps — no prompt fix can address them

## 3. Dense vs MoE Models

### 27B Dense at 32K context: NOT VIABLE
- `inspect_files(256 files)` — model dumps entire index in one tool call, blowing context
- Context-shift produces 0-char outputs on extraction calls
- Single plan took 69 minutes then crashed LM Studio
- Tool caps (inspect_files max 15, read_files max 10) fixed the crash but plan still took 45 min

### Root cause
The 30B MoE (3B active params) prefills fast even on bloated prompts. The 27B dense chokes because ALL 27B params are active on every token. Same prompts, same context window — fundamentally different performance characteristics.

### Lesson
The pipeline was designed for MoE models. Dense models at 32K need either 65K context (not viable on 32GB VRAM) or aggressive prompt compression that defeats the purpose.

## 4. Reasoning Quality Benchmark

### Setup
- Task: "Add query result streaming" (ideal plan by Claude Sonnet as ground truth)
- Scoring: 5 dimensions × 10 points = 50 max (architecture choice, key files, integration points, scope calibration, actionability)
- 10 runs per iteration, scored by Claude Opus against the ideal

### Monolithic pipeline (3 stages)

40 plans across 4 iterations, all fixes regressed:

| Iteration | Change | Score |
|-----------|--------|-------|
| Baseline | None | **13.7/50** |
| Fix #1 | Softer impl check | 10.7 |
| Fix #2 | Focused investigation questions | 10.2 |
| Fix #3 | Tool-augmented investigations | 11.7 |

**Universal failures (0/40 plans):** No plan found Generator Wrapper pattern, synthesizer.py, governance-before-generation, or ABSTAIN handling. Conclusion: model capability ceiling for monolithic reasoning.

### Decomposed pipeline (decision-based)

Replaced 3-stage monolithic with: Call Graph → Decision Decomposition → Per-Decision Resolution → Synthesis.

| Version | Change | Score (30B MoE) |
|---------|--------|-----------------|
| v1 | Initial decomposed pipeline | 16.2/50 |
| v2 | Full import graph + seed files for call graph BFS | 14.5/50 |
| v3 | Annotated call graph edges with method signatures | 15.3/50 |
| **v4** | **Contract preservation rules in resolution prompt** | **18.6/50** |

Each fix addressed a specific failure: v2 found synthesizer.py (0→4/5), v3 added generate_stream (0→3/5), v4 stopped plans from breaking existing method signatures (0→5/5 preserved).

### Signal progression across versions

| Signal | Monolithic | v1 | v2 | v3 | v4 |
|--------|-----------|----|----|----|----|
| synthesizer found | 0/10 | 0/6 | 4/5 | 5/5 | **5/5** |
| generate_stream | 0/10 | 0/6 | 0/5 | 3/5 | **4/5** |
| governance aware | 0/10 | 0/6 | 0/5 | 1/5 | **4/5** |
| answer() preserved | 0/10 | ~5/6 | ~3/5 | 2/5 | **5/5** |
| ABSTAIN | 0/10 | 0/6 | 0/5 | 0/5 | 0/5 |
| StreamingAnswer | 0/10 | 0/6 | 0/5 | 0/5 | 0/5 |

### Haiku via OpenRouter (model capability test)

Used the same decomposed pipeline but with Claude Haiku 4.5 instead of the 30B MoE:

| Model | Avg Score | Best Plan |
|-------|-----------|-----------|
| 30B MoE decomposed v4 | 18.6/50 | 22.0 |
| **Haiku decomposed** | **28.9/50** | **31.0** |

Haiku hit EVERY signal the 30B missed: answer_stream 5/5, ABSTAIN 5/5, StreamingAnswer 3/5. The decomposed pipeline extracts the model's full capability — the model itself is the bottleneck for the 30B.

### Decomposition validator (reverted)
Built a pure-Python validator that injects synthetic decisions for type mismatches and gating gaps. Result: helped 30B slightly but hurt Haiku (28.9→24.2). The bridge question's framing over-constrained downstream reasoning. Design is sound but needs different question framing.

## 5. Key Architectural Decisions

### Call graph extraction is the highest-impact addition
Pure Python, no LLM, ~1 second. Traces caller→callee chains from the structural index. Adding method-level annotations (`engine.answer() → synthesizer.generate()`) directly caused plans to find the right intervention point.

### Contract preservation as a downstream constraint
Adding "never modify existing public method signatures" to the resolution prompt, combined with constraint propagation, eliminated the most common architectural error (modifying `answer()` instead of adding `answer_stream()`).

### The gap between 18.6 and 28.9 is model capability
The pipeline extracts everything the 30B can give. ABSTAIN handling, StreamingAnswer collector, and multi-layer propagation require reasoning the 30B MoE (3B active) cannot do. When better local MoE models release, the pipeline will automatically produce better plans.

### The gap between 28.9 and 50 is pipeline limitations
Even Haiku couldn't produce: code artifacts (no working StreamingAnswer implementation), effort estimates, or properly scoped plans (all inflated to include API/SSE layers). These are synthesis-stage prompt and extraction issues, not model issues.

## 6. Infrastructure Learnings

### Two config files
Windows Store Python creates a second config at `AppData\Local\Packages\PythonSoftwareFoundation.Python.3.12_...\LocalCache\Local\fitz-graveyard\...`. Both must be updated when changing models.

### LM Studio auto-load
LM Studio's JIT model loading (`unloadPreviousJITModelOnLoad`, `autoLoadBundledLLM`) can reload the wrong model mid-benchmark. The config's `model` field tells the API what name to send, but LM Studio serves whatever is loaded regardless.

### OpenRouter integration
Added `api_key` support to `LMStudioClient` — same OpenAI-compatible client works with OpenRouter by changing `base_url` and adding the key. Cost: ~$0.70/plan with Haiku (15 decisions × large output). Too expensive for iteration but useful for capability testing.

### Haiku via subagents (not implemented)
Discussed using Claude Code subagents instead of OpenRouter for Haiku calls — free under subscription. Not practical because the pipeline makes ~20 sequential LLM calls per plan with constraint propagation between them. Would need to replicate the full pipeline loop inside a single agent.

## 7. What's Next

### Short-term (pipeline quality)
- Fix synthesis prompt to demand code artifacts and effort estimates (the 28.9→50 gap)
- The decomposition validator concept is right but needs better question framing
- Consider capping decisions at ~8 (Haiku generated 15, causing $0.70/plan costs and 12-min runtimes)

### Medium-term (model improvements)
- When Qwen3.5-35B-A3B or similar MoE with better reasoning releases, the pipeline automatically improves
- The benchmark (40 queries + Sonnet scoring) is ready to measure any model upgrade instantly

### Architecture direction
- The user proposed a more radical decomposition: separate "what needs deciding" from "decide each thing" with explicit constraint propagation. This IS the decomposed pipeline we built. Further refinement should focus on the synthesis stage, not more decomposition.
