## Architecture

### Explored Approaches
**1. Single-Service Monolithic Pipeline**
- Description: A single Python service implementing the complete planning pipeline with core orchestrator, stage executors, data flow handler, agent context gatherer, and checkpoint system.
- Pros: Minimal operational overhead since all logic resides in one package with clear internal interfaces
- Cons: Teams often over-engineer monoliths by trying to make them modular or service-like when they should remain a focused pipeline

**2. Modular Pipeline Services Architecture**
- Description: Split core pipeline functions into separate microservice components with context service, planning engine service, and data storage service.
- Pros: Makes it easier for other teams in an ecosystem to integrate with specific planning functionality
- Cons: Significant operational overhead due to multiple deployments, service discovery mechanisms, circuit breakers, and inter-service monitoring

### Recommended: Single-Service Monolithic Pipeline

The Command-Line Tool with State Management approach is right for this project because: 1. The scope of the system is well-defined and focused on a planning pipeline producing structured technical documentation, which aligns precisely with command-line tool capabilities; 2. Checkpointing implies an expectation of long-running pipelines that may require resumption after failure or interruption — this fits naturally into a CLI workflow using SQLite storage; 3. There are no explicit requirements for cross-team integration or inter-service communication at this stage, so modular services add unnecessary complexity. Approach A (Monolithic) might be overkill if the intent is to eventually split this into multiple services later and doing it now creates architectural rigidity and introduces unnecessary technical debt without real benefit.

*This task produces one configuration file (`fitz_graveyard/config/planning_config.py`) that defines the schema and default settings for controlling pipeline behavior. It also prepares documentation about how the output structure maps to final plan sections (`docs/planning_output_format.md`). The engineering effort is low, requiring precise understanding of existing schemas but not complex logic or system-level design.*

## Design

### Architectural Decision Records
**ADR: ADR: Single-Service Monolithic Pipeline Approach**
- Decision: Choose the Command-Line Tool with State Management approach as primary implementation strategy for production use cases.
- Rationale: This approach aligns well with project scope, requires minimal operational overhead, supports checkpointing natively via SQLite integration, and avoids unnecessary complexity from microservices or distributed architectures. It also enables rapid iteration without service discovery complications.
- Alternatives: Microservices Architecture — rejected because it introduces significant complexity in service coordination, monitoring, and deployment infrastructure which is not required by this project's constraints., Monolithic Service Approach — rejected because while similar in design, it does not leverage command-line usability advantages of the CLI-based tooling.

**ADR: ADR: Handling Invalid JSON from LLM**
- Decision: Fail loudly rather than silently attempt recovery when parsing fails with `_repair_truncated_json` or `extract_json` methods in base pipeline stages.
- Rationale: This ensures early detection of LLM generation issues, preventing downstream errors from malformed data and encouraging developers to investigate root causes instead of masking them through graceful degradation.
- Alternatives: Silent fallback behavior — rejected because it masks underlying data quality problems in LLM generation which could lead to inconsistent outputs downstream., Attempt auto-repair with retries — rejected since this introduces ambiguity about the actual correctness of generated content and delays debugging.

**ADR: ADR: Configuration Boundary for Pipeline Stages**
- Decision: Support configurable list of `PipelineStage` instances in `fitz_graveyard/planning/pipeline/stages/__init__.py` via configuration schema controls.
- Rationale: This provides flexibility for different use cases and environments where some stages might be skipped or reordered. The system relies on a simple factory function that returns ordered stage list, making this change backward compatible with existing code paths.
- Alternatives: Hardcoded stages in pipeline — rejected because it makes the system inflexible and unable to support dynamic configurations required by varying environments., Plugin architecture with registration mechanism — rejected due to added complexity not justified by current project scope.

**ADR: ADR: Data Consistency Across Pipeline Stages**
- Decision: No external context is maintained between pipeline steps beyond what's defined by the schemas (context, architecture, roadmap). All inter-stage communication occurs through explicitly validated Pydantic model fields.
- Rationale: This prevents leakage of intermediate results and maintains encapsulation. It also ensures type safety at runtime with pydantic validation ensuring data integrity across boundaries.
- Alternatives: Global state management — rejected because it introduces tight coupling and makes the pipeline harder to reason about., Shared memory store for cross-stage knowledge — rejected due to complexity of synchronization, potential race conditions.
