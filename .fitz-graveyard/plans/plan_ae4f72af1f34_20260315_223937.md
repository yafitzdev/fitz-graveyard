## Architecture

### Explored Approaches
**1. Modify LLM Client Signatures**
- Description: Change generate() in fitz_graveyard/llm/client.py, lm_studio.py, and llama_cpp.py to return tuple[str, dict] containing token usage metadata directly.
- Pros: Guaranteed per-call accuracy without race conditions, Simplifies pipeline aggregation logic by threading data explicitly, Eliminates dependency on opaque drain_call_metrics() schema
- Cons: Breaking change requiring updates to all callers like PipelineStage.execute, Higher testing burden across three client implementations, Requires immediate DB migration for SQLiteJobStore persistence

**2. Enhance Metrics Correlation**
- Description: Keep generate() returning str, inject unique request_id into calls, match against drain_call_metrics() output post-execution.
- Pros: Lower immediate code churn in client signatures, Preserves existing async flow structure without refactoring callers
- Cons: High risk of correlation errors in concurrent pipelines due to batching, Metrics schema is currently opaque/unverified per Data Flow Map, Potential data loss if drain timing mismatches stage completion

### Recommended: Modify LLM Client Signatures

This approach solves the root cause identified in the Type Boundary Audit where token data is discarded at the client layer. The alternative relies on probabilistic correlation via drain_call_metrics(), which the Feasibility Report flags as high risk for inaccurate reporting due to batching and lack of unique IDs. Accurate token tracking requires deterministic data flow, not post-hoc metric matching.

## Design

### Architectural Decision Records
**ADR: ADR: Modify LLM Client generate() Signatures to Expose Token Usage**
- Decision: Change the return type of async generate() in all three LLM client classes to tuple[str, dict] or a structured response object containing text and usage_metadata fields (input_tokens, output_tokens).
- Rationale: Direct return guarantees per-call accuracy without race conditions. The Feasibility Report confirms drain_call_metrics() has correlation gaps for specific pipeline stages due to batched returns lacking unique request IDs.
- Alternatives: Rely on drain_call_metrics() — rejected because schema is unknown/unverified and correlation with specific stage execution is probabilistic, Add separate callback hook for token reporting — rejected as more complex than direct return and introduces async coordination overhead

**ADR: ADR: Extend StageResult and PipelineResult Schemas for Token Aggregation**
- Decision: Add optional tokens_used: int | None field to StageResult and total_tokens: int plus stage_token_breakdown: dict[str, int] fields to PipelineResult dataclasses.
- Rationale: Enables end-to-end cost visibility per job. Aligns with existing diagnostic rendering patterns in fitz_graveyard/cli.py where Rich tables display timing and diagnostics alongside output.
- Alternatives: Create separate TokenMetrics object passed alongside results — rejected to avoid coupling complexity and maintain single source of truth in result objects, Store tokens only in logs — rejected as not queryable or persisted for historical analysis

**ADR: ADR: Persist Token Usage Metrics in SQLite Job Store**
- Decision: Add prompt_tokens, completion_tokens, and total_tokens INTEGER columns to the jobs table via database migration script before enabling tracking features.
- Rationale: Enables long-term tracking of costs per job; supports replay and audit features without re-running jobs. Historical data can be set to NULL or 0 during migration.
- Alternatives: Store in separate metrics table — rejected to keep job context unified and simplify queries for cost reporting, Do not persist tokens — rejected as violates requirement to see how many tokens each query costs over time
