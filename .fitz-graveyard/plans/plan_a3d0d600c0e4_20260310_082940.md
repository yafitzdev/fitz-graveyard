## Architecture

### Explored Approaches
**1. Monolithic Pipeline with Hybrid Retrieval Agent**
- Description: This approach implements a single monolithic pipeline that orchestrates all steps through `PlanningPipeline` class. The core agent-based context gathering uses hybrid retrieval techniques from `AgentContextGatherer` which combines BM25, embedding recall, and structural scanning.
- Pros: Clear sequential processing flow, Standardized stage interfaces via PipelineStage base class, Consistent use of hybrid retrieval methods within AgentContextGatherer
- Cons: Tight coupling between stages, High operational burden due to cascade failures, Difficult to add monitoring or concurrent processing without major refactoring

**2. Layered Microservice Architecture**
- Description: This approach decouples context gathering from planning stages into separate services: a dedicated `AgentService` for context retrieval, an `ArchitecturePlanningService` for architecture decisions, and a `RoadmapRiskService` for roadmap generation.
- Pros: Independent scalability of each service, Team autonomy in working on specific layers
- Cons: Excessive complexity in managing multiple services, Over-engineering tendencies without clear benefit at current scale, Complex inter-service communication protocols required

**3. Event-Driven Pipeline with Asynchronous Stage Execution**
- Description: This uses an event-driven architecture where stages communicate via messages or events, with a `PlanningCoordinator` managing the lifecycle and separate services handling processing asynchronously.
- Pros: Easy to add new stages without affecting existing logic, Better parallelization potential
- Cons: Overcomplicated event routing can cause race conditions, Performance bottlenecks from wrong message broker selection

### Recommended: Monolithic Pipeline with Hybrid Retrieval Agent

The monolithic approach is correct for this project's scope because it aligns directly with the codebase design - all core logic lives within a single pipeline, stages are defined via inheritance from PipelineStage, and data flows strictly sequentially. The existing AgentContextGatherer already encapsulates hybrid retrieval mechanisms in one class which maintains consistency rather than introducing unnecessary complexity. While layered microservice architecture would offer scalability benefits, it's premature for this MVP-like design, and event-driven approach introduces overcomplicated asynchronous workflows that don't add value at this point.

- Rationale: This decision maintains consistency with existing hybrid retrieval architecture already present in the codebase. It ensures high-quality file selection that supports accurate LLM understanding while keeping dependencies on external tools minimal.

**ADR: ADR: All LLM calls use fallback mechanisms defined in client.py**
- Decision: All LLM interactions will automatically attempt fallback models when primary model fails due to timeouts or other issues.
- Rationale: This approach ensures robustness against intermittent failures in the LLM backend, especially important given how sensitive structured outputs can be to slight variations in generated text. It matches existing fallback infrastructure already built into client.py utilities.
- Alternatives: Implementing custom retry logic - rejected because it would duplicate functionality already present in client.py's generate_with_fallback method and increase maintenance overhead.

**ADR: ADR: Pipeline stages are strictly executed in sequence with no parallelization**
- Decision: Each pipeline stage will execute one after another without any form of concurrency or parallel processing.
- Rationale: Maintains the simplicity and predictability expected from current implementation. Sequential execution prevents complex synchronization issues and simplifies debugging since each step's output is clearly dependent on previous steps which aligns with how stages are currently structured in PipelineStage base class.
- Alternatives: Adding parallel stage processing - rejected because it would complicate the control flow and introduce unnecessary complexity at this early development stage where correctness is more important than speed.

**ADR: ADR: The agent's hybrid retrieval methods use defined thresholds for recall quality**
- Decision: Enforce maximum caps on candidates during reranking (`_RERANK_TOP_K = 20`) and filter zero-score files from BM25 scoring.
- Rationale: This provides a balance between recall quality and computational overhead. The defined thresholds prevent overloading the LLM with too many irrelevant files while ensuring that high-quality matches are not prematurely filtered out due to overly strict criteria.
- Alternatives: Using unbounded thresholds - rejected because it could lead to inefficient LLM usage and unpredictable performance characteristics especially when dealing with large repositories or complex projects.
