# Project: Create an OpenAI-compatible YAML chat plugin specification that exposes existing

## Context

**Description:**
Create an OpenAI-compatible YAML chat plugin specification that exposes existing FastMCP tools from fitz_graveyard via an OpenAI-style interface, mapping MCP tool signatures to OpenAPI-style schema in a discoverable format.

**Requirements:**
- Generate OpenAI YAML plugin specification from existing MCP tools
- Expose all implemented tools (create_plan, check_status, get_plan, etc.) via standard OpenAI discovery protocol

**Constraints:**
- Must not rebuild what's already implemented in fitz_graveyard/server.py and tools/
- No changes to core tool logic or server lifecycle unless required for compatibility

**Stakeholders:**
- Developer using OpenAI agents
- Maintainer of fitz-graveyard backend
- End-users of fitz-ai system

**Existing Files:**

## Architecture

### Explored Approaches
**1. Minimalist YAML Generator**
- Description: A Python script that parses existing fitz_graveyard/tools/*.py to extract function signatures and converts them into an OpenAPI-style YAML spec compatible with the OpenAI plugin format.
- Pros: Minimal operational burden - just run once to generate the spec, Direct mapping of existing tools to OpenAI plugin format, Leverages already-implemented tool signatures
- Cons: Tool signature mismatches or incorrect parameter type mappings could cause failures, Requires careful handling of Python function introspection

**2. Full Plugin Server Abstraction Layer**
- Description: A new middleware layer that wraps FastMCP with a dedicated OpenAI-compatible HTTP endpoint, handling all routing and parameter translation.
- Pros: Future extensibility into full plugin server if needed
- Cons: Introduces unnecessary complexity when all required tool definitions already exist, Higher operational burden - additional service to maintain, No requirement for a server or runtime component - only spec file generation is needed

**3. Auto-Spec Generator Service**
- Description: A background process or microservice that watches tool changes and auto-generates spec files.
- Pros: Continuous synchronization of specs with code
- Cons: Unnecessary complexity for one-time generation task, Adds maintenance overhead, Overcomplicates a one-time static configuration problem

### Recommended: Minimalist YAML Generator

Approach 1 is correct because: (1) The original request asks to 'build an openai yaml chat plugin' and the context document confirms that all necessary tools are already implemented in fitz_graveyard/tools/, (2) No new services or complex patterns are required since we're only generating a static spec file, (3) There's no indication of needing runtime behavior changes or dynamic loading beyond static spec generation, and (4) The core functionality is already present; this is purely about converting it into OpenAI plugin format.

## Design

### Architectural Decision Records
**ADR: [Scope Exclusion] v1 of this spec does not include dynamic schema discovery features**
- Decision: Omit dynamic schema discovery from the initial spec generation.
- Rationale: This avoids unnecessary complexity and aligns with the static nature of the OpenAI plugin format. The tradeoff is that clients must manually update the spec if tools change, but this prevents over-engineering.
- Alternatives: Include dynamic schema discovery — rejected because it introduces unnecessary complexity for a static deployment scenario

**ADR: [Capability Boundary] Tool signatures in `fitz_graveyard/tools/*.py` are assumed to be compatible with standard OpenAPI schema mapping**
- Decision: Assume compatibility of existing tool signatures with OpenAPI schema mapping.
- Rationale: This leverages the existing implementation and avoids rework. The tradeoff assumes no edge cases in parameter types that might require special handling.
- Alternatives: Implement custom parameter type handling for complex cases — rejected because existing tools are already implemented with standard types

**ADR: [Edge Case Handling] Only successful tool calls are mapped — error handling is omitted from schema**
- Decision: Focus the schema on successful tool call parameters, omitting error responses.
- Rationale: This simplifies the initial schema and aligns with OpenAPI best practices. The tradeoff is that error handling must be documented separately in documentation or code comments.
- Alternatives: Include comprehensive error responses in schema — rejected because OpenAPI spec focuses on success paths and errors are handled at runtime
