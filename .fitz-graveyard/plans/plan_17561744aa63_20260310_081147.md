# Project: FitzPlanner is a system for generating comprehensive software development plans 

## Context

**Description:**
FitzPlanner is a system for generating comprehensive software development plans through LLM-based reasoning and structured prompting, taking natural language job descriptions and producing multi-stage analyses including context understanding, architecture exploration, design decisions, implementation roadmap with phases, and risk assessment.

**Requirements:**
- System architecture uses a pipeline of stages: context → architecture → roadmap/risk
- LLM integration leverages clients for reasoning (Ollama, LM Studio, Llama.cpp)
- Structured output schema defines Pydantic-based models for all plan sections
- Context awareness - system must understand existing codebase structure
- Multi-stage validation logic where each stage validates its own outputs or collaborates with others
- Progress tracking and checkpointing for long-running jobs

**Constraints:**
- Performance: Must not run out-of-memory during generation (enforced by RAM monitoring)
- LLM resource usage: Requires careful orchestration to avoid hitting model limits or timeouts
- Modularity: Pipeline must be extensible allowing new stages to be added without breaking existing flow

## Architecture

### Explored Approaches
**1. Monolithic Single-Process System**
- Description: Core logic in one executable using asyncio, SQLite checkpoints, structured output schemas through Pydantic.
- Pros: Simple deployment and fast inter-stage communication
- Cons: Limited scalability beyond single machine

**2. Service-Oriented Architecture**
- Description: Broken into microservices communicating via gRPC/REST or message queues.
- Pros: Scalable and modular
- Cons: Complex deployment and network overhead

**3. Event-Driven with State Machines**
- Description: Uses event stores for tracking stage lifecycle.
- Pros: Excellent audit trail and replay capabilities
- Cons: High implementation complexity

### Recommended: Monolithic Single-Process System

The monolithic single-process approach is recommended because it aligns with the actual implementation documented in the codebase. The system uses asyncio for async I/O, structured output schemas via Pydantic models (like PlanOutput), and checkpointing through SQLite to support recovery from failures. It does NOT use dynamic provider selection or service-oriented decomposition as suggested in other approaches. Other approaches like Service-Oriented Architecture and Event-Driven with State Machines add unnecessary complexity for a system designed to be self-contained with limited scalability needs, and are not implemented according to the source code.

## Design

### Architectural Decision Records
**ADR: ADR: LLM Provider Integration Strategy**
- Decision: Uses direct concrete implementations from fitz_graveyard.llm.client instead of generic abstract base classes for LLM clients
- Rationale: This approach provides predictable behavior and avoids the complexity of dynamic provider selection while maintaining clear dependencies on specific client implementations that are well-tested in the codebase.
- Alternatives: Generic abstract base class for LLM clients — rejected because it would add unnecessary abstraction layers and complexity without clear benefits given the system's single-provider focus

**ADR: ADR: Context Gathering Integration**
- Decision: Conditional integration with AgentContextGatherer that is enabled/disabled based on configuration and provides file scanning capabilities through BM25 + embedding recall techniques
- Rationale: This approach allows for flexible deployment where context gathering can be disabled when not needed, while still providing a powerful mechanism for local code analysis via the AgentContextGatherer.
- Alternatives: Fully integrated context gathering without conditional logic — rejected because it would make the system less flexible for different deployment scenarios and potentially resource-intensive

**ADR: ADR: Memory Management**
- Decision: Aborts LLM generation when system RAM usage exceeds configured percentage threshold instead of implementing complex fallback strategies
- Rationale: This simple but effective approach prevents the system from crashing due to out-of-memory errors by proactively aborting during high-memory operations, which is suitable for a single-process architecture with limited scalability requirements.
- Alternatives: Implementing complex adaptive memory management strategies — rejected because it would add unnecessary complexity and potential reliability issues for a system designed with limited scalability needs

**ADR: ADR: Staging Order Enforcement**
- Decision: Enforces strict execution order of pipeline stages through hard-coded sequence rather than dynamic configuration or runtime modifications
- Rationale: This ensures deterministic behavior and predictable data flow between stages, which is essential for maintaining the integrity of generated plans across different sections like context, architecture, roadmap/risk, etc.
- Alternatives: Allowing dynamic stage ordering — rejected because it would complicate the pipeline orchestration logic without clear benefits for the system's use case

**ADR: ADR: Checkpointing Model**
- Decision: Uses SQLite-based checkpoint persistence to maintain state during pipeline execution for resumption after interruptions or failures
- Rationale: This provides a simple, reliable mechanism for recovering from partial executions without requiring complex distributed state management that would be overkill for this single-process application architecture.
- Alternatives: Distributed event-sourced state management — rejected because it would introduce significant complexity for a system designed to be monolithic and self-contained without scaling needs
