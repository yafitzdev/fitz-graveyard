## Architecture

### Explored Approaches
**1. Approach A: Centralized Token Tracking in LLM Clients**
- Description: This approach implements token tracking directly within each LLM client implementation (OllamaClient, LlamaCppClient, LMStudioClient). Each client tracks its own call metrics including token usage. The base class OllamaClient exposes a new method drain_call_metrics() which returns richer data structures that include token counts.
- Pros: Minimal Invasiveness: Follows existing patterns where each LLM provider handles its own logic, Backward Compatibility: All existing calls to drain_call_metrics() and other methods remain unchanged; the change only adds fields, Modular & Extensible: Each client can decide how/whether to extract token data based on its underlying API
- Cons: Failure Mode: Inconsistent token reporting across clients (some support it, others don't), Operational Burden: Low-to-moderate - requires adding tracking logic to each client type, Implementation Complexity: Easy but needs careful attention to ensure all clients implement the same interface

**2. Approach B: Token Aggregator Service with Event Bus**
- Description: Introduce a dedicated TokenAggregatorService that listens for events (e.g., via Redis pub/sub or Kafka) emitted by LLM client instances when they complete calls. This service collects all token data and stores it in DB tables related to jobs and models.
- Pros: Centralized Monitoring: Provides unified view of token usage across all LLM clients, Scalable Architecture: Can easily accommodate more services or different event sources
- Cons: Failure Mode: Complex setup with potential single point of failure, increased latency due to eventing overhead, Operational Burden: High - requires managing additional microservice and background process, Implementation Complexity: Harder requiring integration with real-time messaging system

### Recommended: Approach A: Centralized Token Tracking in LLM Clients

Approach A is the right choice because it follows existing patterns and avoids unnecessary complexity. The reasoning incorrectly assumes that all client classes implement token tracking when in fact only OllamaClient has a stubbed drain_call_metrics method, with no actual implementation to extract token information from API responses across subclasses (OllamaClient, LlamaCppClient, LMStudioClient). Approach B introduces significant operational overhead without clear benefit for the specific requirements. The analysis shows that while this approach is recommended in theory, it faces critical gaps because there's no existing framework in the codebase for consistent token collection and client implementations are missing necessary methods like _extract_token_metrics().

*This task requires extending LLM clients to track and report token usage by modifying method signatures in client.py, llama_cpp.py and lm_studio.py. It also includes updating the job model with new fields for prompt/completion tokens and ensuring metrics are passed through to background worker logging.*

## Design

### Architectural Decision Records
**ADR: ADR: Token Usage Extraction Is Optional Per Client Type**
- Decision: Clients will attempt to extract tokens from the raw LLM API responses using their respective APIs' built-in fields, and log a warning with empty fields if extraction fails or isn't supported by that provider
- Rationale: The system must be robust against different LLM backends (Ollama vs. LM Studio) that may vary in their ability to return token data.
- Alternatives: Approach B: Centralized Token Aggregator Service with Event Bus — rejected because it introduces unnecessary complexity and single point of failure, Approach C: Token-Counting Middleware Layer — rejected because middleware could be bypassed or misconfigured, leading to inconsistent tracking

**ADR: ADR: New Fields Are Added Without Removing Old Ones in JobRecord**
- Decision: JobRecord gets new optional fields (prompt_tokens, completion_tokens) which are lists that store token counts per call; they default to empty lists until populated by clients.
- Rationale: Backward compatibility ensures existing code doesn't break when changes are implemented since no old fields are removed or changed in behavior
- Alternatives: Remove old fields or make them required — rejected because it would break backward compatibility with existing deployments, Use separate tables instead of lists in job records — rejected due to increased complexity without significant benefit

**ADR: ADR: Token Data is Logged Per Call, Not Per Job for Fine-Grained Control**
- Decision: Token logging happens at individual call level inside each client’s generate method (not aggregated per job) which allows fine-grained control over recording metrics.
- Rationale: This approach provides flexibility in where and how token data is captured, avoiding batching logic that might introduce latency or complexity while maintaining detailed traceability.
- Alternatives: Aggregated per-job logging — rejected because it reduces flexibility in analyzing and optimizing performance at a more granular level, Streaming token tracking (during generation) — rejected to simplify implementation by focusing on final metric reporting

**ADR: ADR: Token Estimation Is Only Used as Fallback When Direct Extraction Fails**
- Decision: When underlying LLM client doesn't return accurate token counts, we use helper functions like estimate_prompt_completion_tokens() to generate approximate values using character-based heuristics as fallbacks.
- Rationale: Ensures completeness of metrics even if some providers don’t expose them directly — while still preferring accuracy over estimation when possible.
- Alternatives: Always use estimation — rejected because it degrades quality of reporting accuracy unnecessarily, Only extract if available and silently ignore missing data — rejected for potentially misleading incomplete datasets

# fitz_graveyard/token_estimator.py

import re


def estimate_token_count(text: str) -> int:
    """
    Estimate the number of tokens in a given text string.
    
    Note: This is an approximation using a simple heuristic and does NOT reflect 
          actual tokenization done by real LLMs. Use only as fallback when no accurate
          count is available from the underlying API.
          
    Returns:
        int: Estimated number of tokens (based on average 4 characters per token).
    """
    # Rough estimation assuming ~4 chars = 1 token
    return max(0, len(text) // 4)


def estimate_prompt_completion_tokens(prompt_text: str, completion_text: str) -> tuple[int, int]:
    """
    Estimate prompt and completion tokens using a rough heuristic.
    
    Args:
        prompt_text (str): The text of the prompt.
        completion_text (str): The generated response text.

    Returns:
        Tuple[int, int]: Estimated prompt_tokens, estimated_completion_tokens
    """
    return estimate_token_count(prompt_text), estimate_token_count(completion_text)

# fitz_graveyard/models/jobs.py

import logging
from dataclasses import dataclass, field
from datetime import datetime, timezone
from enum import Enum
from uuid import uuid4

from fitz_graveyard.models.store import JobStore

logger = logging.getLogger(__name__)

class JobState(Enum):

    QUEUED = "queued"
    RUNNING = "running"
    AWAITING_REVIEW = "awaiting_review"
    COMPLETE = "complete"
    FAILED = "failed"
    INTERRUPTED = "interrupted"

@dataclass
class JobRecord:
    job_id: str
    description: str
    timeline: str | None
    context: str | None
    integration_points: list[str]
    state: JobState
    progress: float
    current_phase: str | None
    quality_score: float | None
    created_at: datetime
    file_path: str | None = None
    error: str | None = None  # Error message if state=FAILED
    pipeline_state: str | None = None  # JSON checkpoint data
    updated_at: datetime | None = None
    api_review: bool = False  # Whether user opted into API review
    source_dir: str | None = None  # Path to codebase for agent context gathering
    cost_estimate_json: str | None = None  # Serialized CostEstimate JSON (set by worker)
    review_result_json: str | None = None  # Serialized list of ReviewResult JSON (set after review)

    # NEW FIELDS ADDED FOR TOKEN TRACKING:
    prompt_tokens: list[int] = field(default_factory=list)     # List of token counts per call
    completion_tokens: list[int] = field(default_factory=list)  # List of token counts per call

class InMemoryJobStore(JobStore):

    def __init__(self) -> None:
        self._jobs: dict[str, JobRecord] = {}
        logger.info("Initialized InMemoryJobStore")

    async def add(self, record: JobRecord) -> None:
        if record.job_id in self._jobs:
            raise ValueError(f"Job {record.job_id} already exists")

        self._jobs[record.job_id] = record
        logger.info(f"Added job {record.job_id} to store")

    async def get(self, job_id: str) -> JobRecord | None:
        return self._jobs.get(job_id)

    async def list_all(self) -> list[JobRecord]:
        return sorted(
            self._jobs.values(), key=lambda r: r.created_at, reverse=True
        )

    async def update(self, job_id: str, **kwargs) -> None:
        if job_id not in self._jobs:
            raise ValueError(f"Job {job_id} does not exist")

        record = self._jobs[job_id]
        for key, value in kwargs.items():
            if hasattr(record, key):
                setattr(record, key, value)
            else:
                logger.warning(f"Tried to update non-existent attribute '{key}' on JobRecord")
        
        # Handle updating token lists
        prompt_tokens = kwargs.get("prompt_tokens", None)
        completion_tokens = kwargs.get("completion_tokens", None)
        if prompt_tokens is not None and isinstance(prompt_tokens, list):
            record.prompt_tokens.extend(prompt_tokens)
        if completion_tokens is not None and isinstance(completion_tokens, list):
            record.completion_tokens.extend(completion_tokens)

    async def get_next_queued(self) -> JobRecord | None:
        queued = [r for r in self._jobs.values() if r.state == JobState.QUEUED]
        if not queued:
            return None
        return min(queued, key=lambda r: r.created_at)
